# Cluster Context: retail-prod-eks-use1 (EKS us-east-1)

This is a **demo cluster** running a synthetic checkout workload. Latency
and errors are generated by `latency-synthesizer` (no real business logic
runs). Tier definitions still apply ŌĆö the agent should treat the services
as if they were production-critical.

## Cluster Facts

- **Cloud:** AWS us-east-1
- **Platform:** Amazon EKS Auto Mode (Kubernetes 1.33)
- **Observability:** AWS Distro for OpenTelemetry (sidecar) ŌåÆ AWS X-Ray;
  CloudWatch metrics under namespace `RetailProd/Services`
- **Namespace:** `shop-prod`
- **AWS Profile:** `fernhub`
- **Slack channel:** `#retail-prod-incidents`

## Workload

Five "services" (a single Python process pretending to be five via OTel
TracerProviders) emit traces matching this call graph:

```
frontendservice
   ŌööŌöĆ checkoutservice (POST /checkout)
        Ōö£ŌöĆ cartservice (GET /cart/{userId})
        Ōö£ŌöĆ productcatalogservice (POST /products/lookup)
        ŌööŌöĆ paymentservice (POST /charge)              ŌåÉ hot path
             ŌööŌöĆ productcatalogservice (POST /reserve)
```

Plus an optional batch pod `inventory-sync-job` that, when scaled to 1,
saturates a node's CPU and (combined with the synthesizer's CPU burn
during `spike`) throttles paymentservice's container. This is the demo's
"real signal" ŌĆö not just a story, an observable
`container_cpu_cfs_throttled_seconds` increment.

## Service Tiers and Priority Classes

### Critical (Priority Class: `payment-critical`, value 1000000)
Disruption fails customer-visible checkout flows. **Never evict, restart,
or disrupt without explicit human approval.**

| Service | Role |
|---|---|
| checkoutservice | Orchestrates checkout |
| paymentservice  | Processes charges |
| frontendservice | Customer entrypoint |

### User-facing (Priority Class: `user-facing`, value 500000)

| Service | Role |
|---|---|
| cartservice           | Cart state |
| productcatalogservice | Product lookup + reservation |

### Background (Priority Class: `low-priority`, value 100)

| Workload | Role |
|---|---|
| inventory-sync-job | Batch reconciliation. Safe to evict at any time. |

## Known Patterns

- **paymentservice slow but not erroring + checkoutservice 504s.** Classic
  noisy-neighbor pattern. Don't blame checkoutservice. Look at:
  - X-Ray: paymentservice trace duration histogram (bimodal during spike)
  - kubectl top pod: paymentservice CPU utilization
  - `container_cpu_cfs_throttled_seconds` for the paymentservice container
  - Co-located pods on the paymentservice node, ranked by priority
- **`inventory-sync-job` running on the same node as paymentservice.** This
  is the recommended eviction target.

## Eviction Order

1. `inventory-sync-job` (low-priority, batch ŌĆö safe)
2. Other low-priority workloads in `shop-prod`
3. **STOP.** Anything user-facing or critical requires explicit approval
   per the [critical-service-protection](../../universal/critical-service-protection/)
   skill.

Prefer `kubectl_delete pod <name> -n shop-prod` over node drain, per the
universal disk-pressure / noisy-neighbor playbooks.
